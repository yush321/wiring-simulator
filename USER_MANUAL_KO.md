# 사용자 매뉴얼 (넘버링 중심)

이 문서는 `wiring-simulator`에서 넘버링 학습/편집 기능을 빠르게 쓰기 위한 실사용 안내서입니다.

## 1) 기본 사용 흐름

1. 상단 `넘버링` 버튼으로 학습 모드 진입
2. 도면의 빨간 영역을 보고 핀 번호 선택
3. `?` 마커 위치에 맞춰 순서대로 입력
4. 단계 완료 시 자동으로 다음 단계 이동

## 2) 관리자 모드에서 넘버링 편집 열기

1. 상단 `관리자` 버튼으로 관리자 모드 ON
2. `넘버링 편집기 (관리자)` 열기
3. `단계 선택`에서 수정할 단계 선택

## 3) 단계 정보 수정 (영역/가이드/방향)

다음 항목은 수정 후 `단계 저장`을 눌러야 반영됩니다.

- 단계 제목
- componentId / pinPreset
- 설명 문구(guide)
- 채점 방향(`가로`, `세로`, `가로+세로`)
- 역순 허용
- 핀 표시 커스텀
- 도면 영역(rect): `영역 선택 시작` 후 드래그

## 4) 문항(정답) 수정/삭제

문항은 `단계 저장`이 아니라 문항 버튼으로 관리합니다.

### 문항 추가

1. `문항 라벨`, `정답 번호` 입력
2. 필요 시 보기(choices) 입력
3. `문항 추가` 클릭

### 문항 수정

1. `문항 선택 (수정/삭제)`에서 대상 문항 선택
2. 라벨/정답/보기 수정
3. `문항 수정` 클릭

### 문항 삭제

1. `문항 선택 (수정/삭제)`에서 대상 문항 선택
2. `문항 삭제` 클릭

## 5) 저장 반영 순서 (중요)

1. 단계 속성을 바꿨으면 `단계 저장`
2. 문항을 바꿨으면 `문항 수정` 또는 `문항 추가/삭제`
3. 마지막에 `학습 반영` 클릭
4. 넘버링 모드에서 실제 동작 확인

## 6) 자주 헷갈리는 포인트

- `단계 저장`은 단계 메타 정보 저장용입니다.  
  문항 정답 수정은 `문항 수정` 버튼을 써야 합니다.
- 편집 후 화면이 예전처럼 보이면 `학습 반영`을 먼저 누르세요.
- `세로/가로`는 입력 안내 방향용입니다.
- 채점은 현재 작성한 정답 순서를 기준으로 동작합니다.

## 7) 문제 발생 시 체크리스트

1. 올바른 단계를 선택했는가?
2. 정답이 의도한 순서로 입력되어 있는가? (예: `8,5`)
3. `문항 수정` 또는 `단계 저장`을 눌렀는가?
4. `학습 반영` 후 다시 테스트했는가?
5. 필요하면 `JSON 내보내기`로 현재 데이터 백업했는가?

## 8) 추천 운영 방법

1. 큰 수정 전 `JSON 내보내기`로 백업
2. 단계별로 수정 후 즉시 테스트
3. 이상 없으면 다음 단계로 진행

추천은 이렇게 바꾸는 겁니다:

입문 평생 29,000원 (기능 제한: 기본 튜토리얼만)

즉 “평생”은 저가 미끼상품으로 두고, 핵심 가치는 구독/기간권에서 회수하는 구조가 안전합니다.

수익화도 지금 생각하신 방향이 맞는데, “아주 일부만 무료”는 너무 인색하면 전환이 떨어질 수 있어서 `충분히 가치가 느껴지는 무료 샘플`은 있어야 합니다. 제 추천은 `무료: 기초 튜토리얼 1~2개 + 공개도면 1개 + 넘버링 1세트 + 팁 기본편`, `유료: 18개 전체 공개도면 + 전체 넘버링 + 단계별 튜토리얼 재생 + 진행도 동기화 + 이후 업데이트`입니다. 콘텐츠 성격상 구독보다 `1회 전체 해금`이 더 잘 맞고, 스토어 상품도 Apple은 비소모성 IAP, Google Play는 non-consumable one-time product가 가장 자연스럽습니다.

가장 현실적인 목표: 10주 (약 2달 반)

1주차: 프로젝트 기반 다지기
목표: 기존 웹 프로젝트를 iOS/Android 앱으로 실행 가능한 상태로 만들기.
작업 내용:
Capacitor 설치 및 설정: 현재 프로젝트에 Capacitor를 추가하고, Xcode(iOS)와 Android Studio(Android)용 프로젝트를 생성합니다.
기본 실행 확인: 에뮬레이터 또는 실제 기기에서 현재 웹앱이 그대로 뜨는지 확인합니다.
Firebase 프로젝트 생성: Firebase에서 새 프로젝트를 만들고, 인증(Authentication)과 Firestore(DB) 서비스를 활성화합니다.
예상되는 어려움: 개발 환경 설정(Xcode, 안드로이드 SDK 등)에서 예상치 못한 에러가 발생하여 시간을 많이 소요할 수 있습니다. 이 단계는 개발 자체보다 환경 설정과의 싸움이 될 수 있습니다.
2주차: 소셜 로그인 연동
목표: 구글/애플 소셜 로그인을 앱에 연동.
작업 내용:
Firebase 인증 SDK를 자바스크립트 코드에 추가합니다.
로그인 버튼 UI를 만들고, 클릭 시 Firebase 소셜 로그인 창이 뜨도록 연동합니다.
로그인 성공 후, 사용자 정보(이름, 이메일 등)를 받아와 화면에 표시하는 기능을 구현합니다.
예상되는 어려움: 각 소셜 로그인(특히 애플)의 개발자 콘솔 설정이 다소 복잡하고 인증서 관련 이슈가 발생할 수 있습니다.
3-4주차: 데이터 분리 및 서버 연동 (핵심)
목표: 무료/유료 콘텐츠를 분리하고, 유료 콘텐츠는 Firebase DB에서 가져오도록 구조 변경.
작업 내용:
데이터 로직 수정: wiring-engine.js 와 modal.js 등에서 모든 데이터를 로컬(APP_TUTORIAL_CONFIG_DEFAULT 등)에서 읽어오던 구조를 수정합니다. 무료 도면(1~3번)은 기존처럼 앱에 포함시키고, 유료 도면은 서버에서 가져오도록 분기 처리를 합니다.
데이터 이전: answers-data.js 등에서 4~18번 도면의 정답, 튜토리얼 데이터를 JSON 형태로 추출하여 Firebase Firestore DB에 업로드합니다.
데이터 로딩 구현: 사용자가 유료 도면을 선택하면, 로그인 상태를 확인하고 Firestore에서 해당 도면 데이터만 fetch 해오는 코드를 작성합니다. (로딩 중임을 알려주는 UI 처리 포함)
이 단계가 2주나 걸리는 이유: 앱의 핵심 데이터 흐름을 바꾸는 가장 중요한 작업입니다. 기존 코드에 대한 깊은 이해를 바탕으로 비동기(async) 데이터 로딩을 적용해야 하므로, 디버깅에 많은 시간이 소요될 수 있습니다.
5-6주차: 인앱 결제(IAP) 연동 (가장 어려운 부분)
목표: 사용자가 앱 내에서 "모든 기능 해제" 상품을 결제할 수 있게 만들기.
작업 내용:
상품 등록: Apple 앱스토어와 Google Play 스토어에 '기간제 패스' 같은 인앱결제 상품을 등록합니다.
결제 플러그인 연동: Capacitor용 인앱결제 플러그인을 설치하고, 구매 요청 및 결과 처리를 구현합니다.
영수증 검증 (서버): 결제가 성공하면, 앱에서 받은 영수증 정보를 Firebase Cloud Function으로 보내 위변조 여부를 검증합니다. (클라이언트에서만 처리하면 해킹에 취약)
권한 부여: 영수증 검증이 성공하면, 해당 유저의 Firestore DB 정보에 isPremium: true 와 같은 플래그를 기록합니다.
예상되는 어려움: 전체 과정 중 가장 복잡하고 시간이 많이 걸리는 단계입니다. 스토어 정책, 샌드박스 테스트, 서버 사이드 영수증 검증 등 생소한 개념이 많고, 각 스토어의 심사 과정에서 반려될 가능성도 있습니다.
7주차: 기기 제한 기능 구현
목표: 한 계정당 최대 2대의 기기에서만 로그인 허용.
작업 내용:
로그인 시 기기의 고유 ID를 가져와 Firestore의 사용자 정보에 저장합니다. (예: devices: ['ID1', 'ID2'])
로그인 시 저장된 기기 목록을 확인하여 2대를 초과하면 "기기 수 제한을 초과했습니다"와 같은 안내 메시지를 보여주고 로그인을 막습니다.
예상되는 어려움: 안정적으로 기기 고유 ID를 가져오는 방법과, 사용자가 기기를 변경하고 싶을 때 관리해주는 정책을 어떻게 구현할지 추가적인 고민이 필요합니다.
8주차: 최종 다듬기 및 출시 준비
목표: 앱 아이콘, 스플래시 스크린 적용 및 스토어 제출 준비.
작업 내용:
앱 아이콘과 실행 시 처음 보이는 스플래시 스크린을 디자인하고 적용합니다.
다양한 기기(아이폰, 갤럭시 등)에서 UI가 깨지지 않는지 최종 테스트를 진행합니다.
앱 스토어에 등록할 스크린샷, 앱 설명, 개인정보처리방침 문서를 준비합니다.
9-10주차 (버퍼 기간): 스토어 심사 및 수정
앱을 스토어에 제출하고 심사를 기다립니다. (보통 1일 ~ 1주일 소요)
심사 과정에서 수정 요청이 오면 해당 부분을 수정하여 다시 제출합니다. 첫 출시에는 보통 1~2번의 심사 반려를 경험할 수 있으므로, 이 기간을 여유롭게 잡는 것이 좋습니다.
